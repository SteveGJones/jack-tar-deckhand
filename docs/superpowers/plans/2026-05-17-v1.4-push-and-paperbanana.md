## 2 — Resolved design decisions

Decisions taken at plan time with defensible defaults. Each lists the rationale; operator may override at PR review. Where genuine choice remains, the decision is escalated to §3.

### 2.1 — How paperbanana is composed (RESOLVED)

**Decision**: Option 4 (skill cross-invocation) + slim Option 2 (new `academic_figure` strategy).

**Rationale**: keeps jack-tar plugins free of pip dependencies on paperbanana; lets operators install paperbanana via its own marketplace path; uses native Claude Code skill-to-skill invocation; the strategy enum gives strategy-map + classifier a clean place to route from.

**Alternatives considered and rejected**:
- *Option 1 — new wrapper plugin `jack-tar-paperbanana`*: too much maintenance overhead for the value; double-install is operator-hostile.
- *Option 3 — MCP server integration*: cross-process complexity not justified; paperbanana's MCP server is for IDE integration, not pipeline routing.

**Operator override**: confirm or pick a different option.

### 2.2 — Cluster B overlap: editorial-mixed-case (#87) vs strap_style (#93) (RESOLVED)

**Decision**: `editorial-mixed-case` register from #87 declares a default `strap_style: prose-sentence` (from #93). The two are not duplicates — #87 is a high-level register preset that bundles many decisions (palette, typography, layout typology, strap style); #93 is a fine-grained brief field. Operators can use #93 standalone for non-#87-register decks.

**Rationale**: registers are composable atop primitive fields. Removing `editorial-mixed-case` from #87 would lose the bundled-default value; making #93's `strap_style` a register-internal-only field would lose the per-deck override path.

**Operator override**: confirm; or decide that `strap_style` should NOT be a top-level brief field at all and live only inside registers.

### 2.3 — Cluster B schema home for register (RESOLVED)

**Decision**: `register` lives in `TalkBrief.preferences.register` (schema-validated enum) AND surfaces in the brief's Section B header. The schema is the load-bearing definition; the brief Section B carries it for narrative-architect's authoring loop.

**Rationale**: schema validation catches typos at brief-save time (the brief-save lint, v0.2 #25); Section B presence makes the register visible to the persona during R3a authoring.

**Operator override**: prefer schema-only OR Section-B-only.

### 2.4 — Cluster C scope: bridge path only OR also deckhand direct pipeline (RESOLVED)

**Decision**: `full_bleed` scale (#88) lands in BOTH paths — bridge enrichment AND deckhand direct pipeline via `build_deck.js`. The `iterate-slide` skill (#89) is deckhand-direct only in v1; bridge integration is v1.5 candidate.

**Rationale**: full-bleed is a slide-layout primitive that operators using either route need. iterate-slide is workflow machinery best validated on the simpler direct pipeline before extending to the bridge's marker-enrich path.

**Operator override**: scope #88 to bridge-only if you want a smaller PR.

### 2.5 — #89 iterate-slide spike answer (RESOLVED via paperbanana research)

**Decision**: borrow paperbanana's `ResumeState` schema + `continue_run(user_feedback)` threading pattern. The skill writes per-run state to `<deck-dir>/.iterate-state.json` with the slide-index, prior prompt, prior verdict, and user critique; subsequent invocations load the state, thread the user_feedback into a regenerate-then-review loop.

**Rationale**: paperbanana already solves this for single-figure work; the contract translates cleanly to single-slide regeneration. Saves the spike investigation.

**Operator override**: confirm or surface concerns about state-on-disk approach.

### 2.6 — #90 composition primitives generalisability (RESOLVED)

**Decision**: v1 ships the 5 primitives named in the issue (two-port fixture, asymmetric towers, multi-craft hub, instrument-grid, three-tier chain-of-authority). v1 also includes a `composition_primitives.schema.json` defining the primitive shape and an `_authoring_guide.md` documenting how operators add new ones. v1 does NOT claim domain-generality; the 5 primitives are aerospace-flavoured per the operator's filing context.

**Rationale**: the primitives in the title are specific; trying to generalise prematurely produces weak abstractions. The schema + guide give the extension path without overcommitting.

**Operator override**: drop the schema if you prefer a more rigid "just 5 primitives" approach.

### 2.7 — #92 Imagen parallel fix (RESOLVED)

**Decision**: yes, fix `_generate_via_imagen` empty-candidates path in the same PR as `_generate_via_nano_banana`. Same bug class, same exception. One PR, two fixes.

**Rationale**: shipping the Nano Banana fix while leaving Imagen unguarded is a known-incomplete patch. The Imagen fix is ~5 additional lines.

**Operator override**: scope to Nano Banana only if you have reason to believe Imagen's empty-candidates path differs materially.

### 2.8 — #92 safety-filter softening vocabulary (PARTIAL)

**Decision**: v1 includes a hard-coded vocabulary of common safety-filter triggers and softer replacements (e.g. "destroy" → "neutralise", "kill" → "stop", "weapon" → "implement"). The full mapping is in `src/safety_filter_vocab.py`. Operators can override the mapping via a `SAFETY_FILTER_VOCAB_PATH` env var pointing at a custom JSON.

**Open question for operator**: scope of the vocab — 20 entries (small focused set) or 60+ (comprehensive)? My recommendation: ship 20 in v1, expand as real-world cases surface.

### 2.9 — Discipline-hook propagation gap (#86) — scope (PARTIAL)

**Decision**: #86 ships in the v1.4 push BEFORE Cluster B / C / Paperbanana so subsequent agent dispatches are protected. Specifically: investigate whether Task subagents inherit `settings.json` hooks; if so, document; if not, add a fallback enforcement mechanism (e.g. `JACK_TAR_BLOCK_PNG_READ=1` env var that Bash wrappers check).

**Open question for operator**: should this be a one-day pre-flight before Cluster A onwards, or land as a parallel track with Cluster D?

## 5 — Sequencing rationale

The order isn't optional — choices ripple through. Here's why this specific order:

1. **#86 first (pre-flight)** — protects every subsequent delegation. Without this, the discipline-hook bypass we discovered in PR #85 will recur in every Task-dispatched agent's work. Land before any other delegated implementation.

2. **Cluster D (#92) second** — substrate hardening. Empty-candidates crash is real, reproducible, and any subsequent dogfood run is at risk of hitting it. Cheap and isolated, paves the way for the more substantial features.

3. **Cluster B (#93 → #87) third** — register vocabulary is a building block. Cluster A's prompt-engineer enhancements lean on register declarations to specialise; without it the prompt-engineer would have to invent its own vocabulary. #93 first because it reconciles the smaller scope first; #87 then composes on top.

4. **Paperbanana integration EPIC fourth** — the `academic_figure` strategy and its imagegen-bridge dispatch land after register vocabulary so the strategy classifier can use register information ("infographic-narrative" → likely composed; "schematic-diagram" → likely academic_figure if paperbanana available).

5. **Cluster A (#91 → #90) fifth** — prompt-engineer enhancements. #91 (text-density warning) lands first — small, mechanical, no design decisions. #90 (composition primitives) on top — uses the register vocabulary (Cluster B) and may reference paperbanana's primitives.

6. **Cluster C (#88 → #89) sixth** — assembler + iterate-slide. #88 (full-bleed scale) is independent of paperbanana but composes with the register vocabulary. #89 (iterate-slide) is the most complex; lands last because it integrates the most surface — register, full-bleed scale, paperbanana resume-state pattern.
